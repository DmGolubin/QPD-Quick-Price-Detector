# Документ требований: Price Monitor Pro

## Введение

Price Monitor Pro — продвинутая система мониторинга цен на товары в интернет-магазинах, аналог distill.io, специализированный на отслеживании цен. Система предоставляет Telegram-бот как основной интерфейс, веб-дашборд с графиками и аналитикой, REST API для интеграций, и архитектурно подготовлена к расширению в виде браузерного расширения и мобильного приложения. Текущий прототип на Python/FastAPI + Playwright + PostgreSQL развёрнут на Railway.

## Глоссарий

- **Система** (System): серверная часть Price Monitor Pro, включающая API, движок мониторинга, планировщик и обработчики уведомлений
- **Монитор** (Monitor): сущность, описывающая отслеживаемый товар — URL страницы, селектор элемента с ценой, условия алертов, интервал проверки и метаданные
- **Движок_Скрапинга** (Scraping_Engine): компонент, отвечающий за загрузку веб-страниц через Playwright, рендеринг JavaScript, извлечение данных из DOM
- **Парсер_Цен** (Price_Parser): компонент, извлекающий числовое значение цены из текстового содержимого элемента страницы
- **Планировщик** (Scheduler): компонент, управляющий расписанием периодических проверок цен для всех активных мониторов
- **Telegram_Бот** (Telegram_Bot): интерфейс взаимодействия с пользователем через Telegram, поддерживающий команды, inline-клавиатуры и conversational flows
- **Веб_Дашборд** (Web_Dashboard): веб-интерфейс для управления мониторами, просмотра графиков и аналитики
- **API_Сервер** (API_Server): RESTful API для программного доступа ко всем функциям системы
- **Условие_Алерта** (Alert_Condition): правило, определяющее когда отправлять уведомление пользователю (порог цены, процент изменения, regex-совпадение)
- **Канал_Уведомлений** (Notification_Channel): способ доставки алерта пользователю (Telegram, Email, Webhook, Discord, Slack)
- **История_Цен** (Price_History): хронологическая запись всех зафиксированных значений цены для монитора
- **Визуальный_Селектор** (Visual_Selector): компонент для визуального выбора элемента на веб-странице через iframe-превью
- **Группа_Сравнения** (Comparison_Group): набор мониторов для одного товара в разных магазинах, позволяющий сравнивать цены
- **Тег** (Tag): метка для организации и фильтрации мониторов
- **API_Ключ** (API_Key): токен аутентификации для доступа к REST API
- **Прокси** (Proxy): промежуточный сервер для маскировки запросов при скрапинге
- **Пользователь** (User): зарегистрированный пользователь системы, идентифицируемый по Telegram chat_id или API-ключу
- **Макрос** (Macro): последовательность автоматизированных действий (клик, ввод текста, прокрутка, ожидание), выполняемых на странице перед извлечением цены
- **Шаг_Макроса** (Macro_Step): одно действие в последовательности макроса с типом, селектором и параметрами
- **Статус_Доступности** (Availability_Status): состояние товара — "в наличии" или "нет в наличии", определяемое по индикаторам на странице
- **Скриншот** (Screenshot): снимок веб-страницы, сделанный при проверке цены, хранимый в формате JPEG
- **Дайджест** (Digest): сводное сообщение с накопленными уведомлениями и аналитикой за период
- **Тихие_Часы** (Quiet_Hours): временной интервал, в течение которого уведомления ставятся в очередь и не отправляются
- **Шаблон_Монитора** (Monitor_Template): предварительно настроенный набор селекторов, валюты и паттернов доступности для известного интернет-магазина
- **Нормализованный_URL** (Normalized_URL): URL после приведения к каноническому виду (удаление trailing slash, сортировка query-параметров, lowercase домена)

## Требования

### Требование 1: Управление мониторами (CRUD)

**User Story:** Как пользователь, я хочу создавать, просматривать, редактировать и удалять мониторы цен, чтобы управлять списком отслеживаемых товаров.

#### Критерии приёмки

1. WHEN Пользователь отправляет запрос на создание монитора с валидным URL, THE Система SHALL создать новый Монитор со статусом "активен" и выполнить первичную проверку цены в течение 10 секунд
2. WHEN Пользователь отправляет запрос на создание монитора с невалидным URL, THE Система SHALL вернуть ошибку с описанием "Невалидный URL: ожидается http:// или https://"
3. WHEN Пользователь запрашивает список мониторов, THE Система SHALL вернуть все мониторы Пользователя, отсортированные по дате создания (новые первыми), включая текущую цену, статус и дату последней проверки
4. WHEN Пользователь отправляет запрос на обновление монитора с валидными данными, THE Система SHALL обновить указанные поля монитора и установить поле updated_at в текущее время
5. WHEN Пользователь отправляет запрос на удаление монитора, THE Система SHALL удалить Монитор и всю связанную Историю_Цен и записи алертов
6. WHEN Пользователь переключает статус монитора, THE Система SHALL изменить поле is_active на противоположное значение и при деактивации прекратить плановые проверки для данного монитора
7. THE Система SHALL хранить для каждого монитора: название, URL, CSS-селектор, XPath-селектор, JavaScript-выражение, пороги уведомлений, валюту, интервал проверки, теги и метаданные создания

### Требование 2: Движок скрапинга и извлечение цен

**User Story:** Как пользователь, я хочу чтобы система надёжно извлекала цены с любых веб-страниц, включая SPA с JavaScript-рендерингом, чтобы получать актуальные данные.

#### Критерии приёмки

1. WHEN Движок_Скрапинга получает URL страницы, THE Движок_Скрапинга SHALL загрузить страницу через Playwright с полным JavaScript-рендерингом и ожиданием до 30 секунд на загрузку
2. WHEN для монитора указан CSS-селектор, THE Движок_Скрапинга SHALL извлечь текстовое содержимое элемента, соответствующего CSS-селектору, с таймаутом ожидания элемента 10 секунд
3. WHEN для монитора указан XPath-селектор, THE Движок_Скрапинга SHALL извлечь текстовое содержимое элемента, соответствующего XPath-выражению
4. WHEN для монитора указано JavaScript-выражение, THE Движок_Скрапинга SHALL выполнить выражение в контексте страницы и использовать результат как исходный текст цены
5. WHEN для монитора не указан селектор, THE Движок_Скрапинга SHALL выполнить автоматическое определение элемента с ценой, перебирая список из не менее 15 распространённых CSS-селекторов цен
6. WHEN страница содержит cookie-баннеры или всплывающие окна, THE Движок_Скрапинга SHALL попытаться закрыть всплывающие элементы перед извлечением цены
7. IF загрузка страницы завершается ошибкой таймаута, THEN THE Движок_Скрапинга SHALL выполнить до 3 повторных попыток с экспоненциальной задержкой (5с, 15с, 45с)
8. IF загрузка страницы завершается HTTP-ошибкой (4xx, 5xx), THEN THE Движок_Скрапинга SHALL записать код ошибки в лог и пометить проверку как неуспешную без повторных попыток для 4xx
9. THE Движок_Скрапинга SHALL ротировать User-Agent из пула не менее 10 различных строк при каждом запросе
10. WHERE настроен Прокси, THE Движок_Скрапинга SHALL маршрутизировать запросы через указанный прокси-сервер

### Требование 3: Парсинг цен и мультивалютность

**User Story:** Как пользователь, я хочу чтобы система корректно распознавала цены в различных форматах и валютах, чтобы отслеживать товары в международных магазинах.

#### Критерии приёмки

1. WHEN Парсер_Цен получает текст с ценой в формате "1 299,90 ₽", THE Парсер_Цен SHALL извлечь числовое значение 1299.90
2. WHEN Парсер_Цен получает текст с ценой в формате "$12.99" или "12,99 €", THE Парсер_Цен SHALL извлечь числовое значение 12.99
3. WHEN Парсер_Цен получает текст с разделителями тысяч (пробел, точка, запятая), THE Парсер_Цен SHALL корректно определить разделитель тысяч и десятичный разделитель
4. THE Парсер_Цен SHALL поддерживать валюты: RUB (₽), USD ($), EUR (€), GBP (£), KZT (₸), CNY (¥), TRY (₺)
5. WHEN Пользователь указывает целевую валюту, отличную от валюты цены на странице, THE Система SHALL конвертировать цену по актуальному курсу, обновляемому не реже одного раза в 6 часов
6. THE Форматировщик_Цен SHALL преобразовать числовое значение цены обратно в текстовое представление с корректными разделителями и символом валюты
7. FOR ALL валидных текстовых представлений цен, парсинг с последующим форматированием и повторным парсингом SHALL вернуть эквивалентное числовое значение (свойство round-trip)

### Требование 4: Система условий и алертов

**User Story:** Как пользователь, я хочу настраивать гибкие условия для уведомлений, чтобы получать алерты только о значимых изменениях цен.

#### Критерии приёмки

1. WHEN цена монитора опускается ниже заданного порога threshold_below, THE Система SHALL отправить уведомление "Цена ниже порога" через все настроенные Каналы_Уведомлений для данного монитора
2. WHEN цена монитора поднимается выше заданного порога threshold_above, THE Система SHALL отправить уведомление "Цена выше порога" через все настроенные Каналы_Уведомлений
3. WHEN цена монитора изменяется на процент, превышающий заданный threshold_pct, THE Система SHALL отправить уведомление с указанием абсолютного и процентного изменения
4. WHEN Пользователь создаёт составное условие с оператором AND, THE Система SHALL отправить уведомление только при одновременном выполнении всех подусловий
5. WHEN Пользователь создаёт составное условие с оператором OR, THE Система SHALL отправить уведомление при выполнении хотя бы одного подусловия
6. WHEN алерт отправлен для монитора, THE Система SHALL не отправлять повторный алерт по тому же условию в течение настраиваемого периода cooldown (по умолчанию 1 час)
7. WHEN цена монитора достигает исторического минимума, THE Система SHALL отправить специальное уведомление "Лучшая цена за всё время"
8. THE Система SHALL записывать каждый отправленный алерт в таблицу alerts_log с типом алерта, сообщением, ценой и временной меткой


### Требование 5: Каналы уведомлений

**User Story:** Как пользователь, я хочу получать уведомления через различные каналы (Telegram, Email, Webhook, Discord, Slack), чтобы не пропустить важные изменения цен.

#### Критерии приёмки

1. THE Система SHALL поддерживать следующие Каналы_Уведомлений: Telegram (основной), Email (SMTP), Webhook (HTTP POST), Discord (webhook URL), Slack (webhook URL)
2. WHEN алерт срабатывает, THE Система SHALL отправить уведомление через все активные Каналы_Уведомлений, настроенные для данного монитора
3. IF отправка уведомления через Канал_Уведомлений завершается ошибкой, THEN THE Система SHALL выполнить до 3 повторных попыток с задержкой 10 секунд и записать ошибку в лог
4. WHEN уведомление отправляется через Telegram, THE Система SHALL форматировать сообщение с HTML-разметкой, включая название товара, старую и новую цену, процент изменения и ссылку на товар
5. WHEN уведомление отправляется через Webhook, THE Система SHALL отправить HTTP POST запрос с JSON-телом, содержащим monitor_id, name, url, old_price, new_price, change_pct и timestamp
6. THE Система SHALL позволять Пользователю настраивать Каналы_Уведомлений на уровне каждого монитора и на глобальном уровне

### Требование 6: Планировщик проверок

**User Story:** Как пользователь, я хочу настраивать индивидуальные интервалы проверки для каждого монитора, чтобы балансировать между актуальностью данных и нагрузкой на систему.

#### Критерии приёмки

1. THE Планировщик SHALL поддерживать интервалы проверки от 60 секунд до 30 дней для каждого монитора
2. WHEN наступает время плановой проверки монитора, THE Планировщик SHALL поставить задачу проверки в очередь выполнения
3. WHILE Монитор имеет статус is_active = false, THE Планировщик SHALL пропускать плановые проверки для данного монитора
4. WHEN Пользователь изменяет интервал проверки монитора, THE Планировщик SHALL применить новый интервал начиная со следующей проверки
5. IF очередь проверок содержит более 100 задач, THEN THE Планировщик SHALL выполнять задачи с ограничением не более 5 параллельных проверок для предотвращения перегрузки
6. THE Планировщик SHALL распределять проверки равномерно во времени, избегая одновременного запуска всех проверок

### Требование 7: Telegram-бот (основной интерфейс)

**User Story:** Как пользователь, я хочу полноценно управлять мониторами через Telegram-бот с удобным conversational UI, чтобы быстро добавлять и контролировать отслеживание цен.

#### Критерии приёмки

1. WHEN Пользователь отправляет команду /add, THE Telegram_Бот SHALL запустить пошаговый диалог: название → URL → селектор (с опцией /skip) → пороги (с опцией /skip) → подтверждение
2. WHEN Пользователь отправляет команду /list, THE Telegram_Бот SHALL отобразить список всех мониторов с inline-клавиатурой для быстрых действий (пауза/возобновление, удаление)
3. WHEN Пользователь отправляет команду /check, THE Telegram_Бот SHALL выполнить немедленную проверку всех активных мониторов и отобразить результаты с изменениями цен
4. WHEN Пользователь отправляет URL без команды, THE Telegram_Бот SHALL предложить создать монитор для данного URL с автоматическим определением цены и названия товара
5. WHEN Пользователь отправляет команду /edit_{id}, THE Telegram_Бот SHALL запустить диалог редактирования монитора с возможностью изменить любое поле
6. WHEN Пользователь отправляет команду /report, THE Telegram_Бот SHALL сформировать сводный отчёт по всем мониторам за последние 24 часа, включая изменения цен и сработавшие алерты
7. WHEN Пользователь отправляет команду /compare_{group_id}, THE Telegram_Бот SHALL отобразить сравнительную таблицу цен для всех мониторов в Группе_Сравнения
8. WHEN Пользователь нажимает inline-кнопку действия, THE Telegram_Бот SHALL выполнить действие и обновить сообщение с актуальным состоянием в течение 3 секунд
9. WHEN Пользователь отправляет команду /settings, THE Telegram_Бот SHALL отобразить текущие настройки (интервал по умолчанию, каналы уведомлений, часовой пояс) с inline-клавиатурой для изменения
10. WHEN Пользователь отправляет команду /export, THE Telegram_Бот SHALL отправить файл с экспортом всех мониторов в формате JSON

### Требование 8: Веб-дашборд

**User Story:** Как пользователь, я хочу иметь веб-дашборд с тёмной темой для визуального управления мониторами и просмотра графиков цен, чтобы анализировать тренды.

#### Критерии приёмки

1. THE Веб_Дашборд SHALL отображать главную страницу со списком всех мониторов, статистикой (общее количество, активные, с ценой, точки данных) и формой добавления нового монитора
2. WHEN Пользователь открывает страницу детализации монитора, THE Веб_Дашборд SHALL отображать график цен (Chart.js) с переключателями периодов (7д, 30д, 90д, 1г, всё), статистику (мин/макс/средняя) и таблицу истории
3. WHEN Пользователь выбирает период на графике, THE Веб_Дашборд SHALL загрузить данные за выбранный период через API и обновить график без перезагрузки страницы
4. THE Веб_Дашборд SHALL поддерживать адаптивный дизайн для корректного отображения на экранах шириной от 320px до 2560px
5. WHEN Пользователь добавляет монитор через веб-форму, THE Веб_Дашборд SHALL выполнить первичную проверку цены и перенаправить на главную страницу с результатом
6. THE Веб_Дашборд SHALL использовать тёмную тему с цветовой схемой: фон #0a0a0f, карточки #1a1a2e, акцент #6c5ce7, зелёный #00b894 (снижение цены), красный #e17055 (рост цены)
7. WHEN Пользователь фильтрует мониторы по тегу или поисковому запросу, THE Веб_Дашборд SHALL отобразить только соответствующие мониторы без перезагрузки страницы
8. THE Веб_Дашборд SHALL отображать страницу истории алертов с фильтрацией по монитору, типу алерта и периоду

### Требование 9: Визуальный селектор элементов

**User Story:** Как пользователь, я хочу визуально выбирать элемент с ценой на странице через веб-интерфейс, чтобы не вводить CSS-селекторы вручную.

#### Критерии приёмки

1. WHEN Пользователь вводит URL в визуальном селекторе, THE Визуальный_Селектор SHALL загрузить страницу в iframe-превью через серверный прокси и отобразить содержимое
2. WHEN Пользователь кликает на элемент в превью, THE Визуальный_Селектор SHALL сгенерировать CSS-селектор для выбранного элемента и подсветить элемент рамкой
3. WHEN Пользователь кликает на элемент, THE Визуальный_Селектор SHALL отобразить панель с: сгенерированным CSS-селектором, XPath-выражением, текстовым содержимым элемента и распознанной ценой
4. WHEN Пользователь нажимает кнопку "Расширить выделение", THE Визуальный_Селектор SHALL переместить выделение на родительский элемент
5. WHEN Пользователь нажимает кнопку "Сузить выделение", THE Визуальный_Селектор SHALL переместить выделение на первый дочерний элемент
6. WHEN Пользователь подтверждает выбор элемента, THE Визуальный_Селектор SHALL сохранить сгенерированный селектор в поле css_selector монитора

### Требование 10: REST API

**User Story:** Как разработчик, я хочу иметь REST API для программного доступа ко всем функциям системы, чтобы интегрировать мониторинг цен в другие приложения и подготовить архитектуру для браузерного расширения.

#### Критерии приёмки

1. THE API_Сервер SHALL предоставлять эндпоинты: GET/POST /api/monitors, GET/PUT/DELETE /api/monitors/{id}, GET /api/monitors/{id}/history, GET /api/monitors/{id}/chart, POST /api/monitors/{id}/check, GET /api/alerts, GET /api/stats
2. WHEN запрос к API не содержит валидный API_Ключ в заголовке Authorization, THE API_Сервер SHALL вернуть HTTP 401 с телом {"error": "Unauthorized"}
3. WHEN запрос к API содержит валидный API_Ключ, THE API_Сервер SHALL выполнить запрошенную операцию и вернуть результат в формате JSON
4. THE API_Сервер SHALL ограничивать количество запросов до 100 запросов в минуту на один API_Ключ и возвращать HTTP 429 при превышении лимита

**User Story:** Как пользователь, я хочу чтобы система продолжала работать при сбоях отдельных компонентов и корректно обрабатывала ошибки, чтобы не терять данные.

#### Критерии приёмки

1. IF соединение с базой данных прерывается, THEN THE Система SHALL выполнить до 5 попыток переподключения с экспоненциальной задержкой и продолжить работу после восстановления соединения

4. WHEN ошибка происходит при обработке запроса API, THE API_Сервер SHALL вернуть структурированный JSON-ответ с полями error и detail, соответствующим HTTP-кодом (400, 404, 500) и не раскрывать внутренние детали реализации

### Требование 26: Уведомления о проблемах с мониторами

**User Story:** Как пользователь, я хочу получать уведомления когда монитор перестаёт работать, чтобы вовремя исправить настройки или URL.

#### Критерии приёмки

1. WHEN Монитор не может извлечь цену в течение 3 последовательных проверок, THE Система SHALL отправить уведомление "Монитор не работает" с описанием ошибки через все настроенные Каналы_Уведомлений
2. WHEN Монитор успешно извлекает цену после серии неудачных проверок, THE Система SHALL отправить уведомление "Монитор восстановлен" через все настроенные Каналы_Уведомлений
3. THE Система SHALL хранить счётчик последовательных неудачных проверок (consecutive_failures) для каждого Монитора
4. THE Веб_Дашборд SHALL отображать статус ошибки и текст последней ошибки на карточке каждого Монитора
5. WHEN Монитор находится в состоянии ошибки более 24 часов, THE Система SHALL автоматически приостановить Монитор (is_active = false) и отправить уведомление Пользователю о приостановке
6. WHEN Пользователь отправляет команду /list, THE Telegram_Бот SHALL отображать индикатор ошибки (⚠️) рядом с мониторами, имеющими consecutive_failures > 0

### Требование 27: Пагинация и производительность при большом количестве мониторов

**User Story:** Как пользователь с большим количеством мониторов, я хочу чтобы система работала быстро и отображала данные постранично, чтобы интерфейс оставался отзывчивым.

#### Критерии приёмки

1. THE API_Сервер SHALL поддерживать пагинацию для эндпоинтов, возвращающих списки, с параметрами: page (по умолчанию 1), per_page (по умолчанию 20, максимум 100)
2. WHEN API_Сервер возвращает ответ с пагинацией, THE API_Сервер SHALL включать объект meta с полями: total, page, per_page, total_pages
3. WHEN Пользователь отправляет команду /list, THE Telegram_Бот SHALL отображать результаты постранично по 10 мониторов на страницу с inline-кнопками "Далее" и "Назад"
4. THE Веб_Дашборд SHALL реализовать бесконечную прокрутку или пагинацию для списка мониторов
5. THE Система SHALL использовать LIMIT/OFFSET с соответствующими индексами базы данных для запросов списков мониторов
6. THE Система SHALL кэшировать часто запрашиваемые данные (список мониторов, статистика) с временем жизни кэша 60 секунд

### Требование 28: Дублирование мониторов и валидация

**User Story:** Как пользователь, я хочу чтобы система предупреждала о дублирующихся мониторах и валидировала входные данные, чтобы не отслеживать один и тот же URL дважды и избежать ошибок конфигурации.

#### Критерии приёмки

1. WHEN Пользователь создаёт Монитор с URL, который уже существует для данного Пользователя, THE Система SHALL предупредить о дубликате и запросить подтверждение перед созданием
2. THE Система SHALL нормализовать URL перед сравнением: удалить trailing slash, отсортировать query-параметры по алфавиту, привести домен к нижнему регистру (Нормализованный_URL)
3. WHEN Пользователь подтверждает создание дубликата, THE Система SHALL создать Монитор (допуская намеренные дубликаты с разными селекторами)
4. THE Система SHALL валидировать синтаксис CSS-селектора перед сохранением Монитора и возвращать ошибку "Невалидный CSS-селектор" при некорректном синтаксисе
5. THE Система SHALL валидировать что значение check_interval находится в допустимом диапазоне (от 60 секунд до 30 дней) и возвращать ошибку при выходе за пределы диапазона
